Projet de fin d'année — Mastère Cybersécurité 2025/2026

ToolboxV8

Plateforme web qui automatise les cinq phases d'un test d'intrusion et génère un rapport PDF prêt à livrer au client. Travail de groupe, rôle : intégration des outils offensifs et QA.

Toolbox automatisée de tests d'intrusion

Rôle dans l'équipe : intégration des outils offensifs & QA
→ github.com/Vyuob/toolbox-m1

L'ergonomie est pensée pour un analyste, pas pour un développeur : un formulaire, un choix de profil, un clic. Chaque outil expose des profils par chips (Quick / Standard / Full…) qui mappent vers la vraie ligne de commande, et la sortie CLI brute est rendue telle quelle dans le rapport PDF final.

01 — Offensif (OSINT)
Recon passive

24 templates de dorks Google / Bing / DuckDuckGo, en 3 catégories : mot-clé, réseaux sociaux, domaine.

02 — Offensif
Recon active

Nmap, résolution DNS, whois, WhatWeb.

03 — Offensif
Scan vulnérabilités

Nmap NSE (--script=vuln), Nikto, SSLyze avec pré-check TCP IPv4/IPv6.

04 — Offensif
Exploitation

SQLmap, Hydra, John the Ripper jumbo (304 formats), Metasploit.

05 — Offensif
Web & API

OWASP ZAP (spider + active, polling auto), Gobuster, Dependency-Check.

FastAPI Celery Kali Rolling PostgreSQL Redis Elasticsearch / Logstash / Kibana MinIO (S3) Caddy (HTTPS) OWASP ZAP RBAC 3 rôles CI/CD GitHub Actions + GitLab CI
architecture

Une stack dockerisée, 3 services applicatifs

Reverse proxy HTTPS, app web séparée de l'API REST, worker Celery exécutant les outils sur une image Kali, infra complète pour le SIEM et le stockage des rapports.
ServiceRôle
caddyReverse proxy HTTPS, certificat auto-signé via CA interne (Let's Encrypt en prod en 1 ligne)
web (FastAPI, :3000)Sert les pages HTML, auth par formulaire + cookie HttpOnly, proxifie /api/*
api (FastAPI, :8000)Endpoints REST purs, accessibles pour intégrations externes (Swagger inclus)
worker (Celery + Kali)Exécute les scans offensifs avec tous les outils préinstallés
db / redis / minioPostgreSQL 16, file d'attente, stockage S3 des rapports PDF
elasticsearch + logstash + kibanaBackend SIEM, visualisation des logs collectés
zapOWASP ZAP en daemon, API REST :8080, polling automatique par le module web_scan
targetCible vulnérable (SSH faible) réutilisable pour les démos Hydra
sécurité & reporting

Pensé pour être livré, pas juste démontré

Sécurité applicative

  • RBAC à 3 rôles (admin, analyst, reader), création auto des comptes au premier démarrage
  • Page de gestion des utilisateurs réservée à l'admin (créer, modifier le rôle, désactiver)
  • Chiffrement Fernet pour les secrets stockés
  • Audit logs append-only en base : login, lancement de scan, génération de rapport, blocage IP

Reporting

  • Rapport PDF généré avec ReportLab : charte pro, header bleu, encart synthèse, sortie CLI brute préservée par outil
  • Vue HTML alignée sur le PDF pour la cohérence consultation/téléchargement
  • SIEM consultable depuis la page /siem (Chart.js) en plus de Kibana

Mon rôle : intégration offensive

Intégration des modules offensifs dans la plateforme : câblage des outils (Nmap, Nikto, SQLmap, Hydra, John, ZAP, Gobuster…) aux profils par chips, gestion des timeouts adaptés par profil, validation des sorties CLI avant rendu dans le rapport.

Mon rôle : QA

Tests fonctionnels sur les modules avant intégration au pipeline CI (lint ruff → pytest avec services PostgreSQL/Redis → build des images Docker), vérification de la cohérence des rapports générés et des cas limites (cibles sans TLS, timeouts dépassés).
